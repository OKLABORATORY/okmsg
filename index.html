Overview

OKMESSAGING is an enterprise-grade SMS and SMPP communication platform engineered for high-volume, secure, and controlled message delivery at telecom scale.

The platform covers the complete lifecycle of SMS communication ‚Äî from infrastructure planning and deployment to SMPP/HTTP message submission, intelligent routing, buffering, protocol translation, billing, and delivery reporting. It follows a strict role-based access control (RBAC) model with clearly separated Admin and User roles, ensuring secure system management and controlled operational usage across all components.

With support for single and bulk SMS, multi-destination messaging, advanced routing logic, Redis-backed high-performance processing, and detailed logs and reports, OKMESSAGING delivers a scalable, resilient, and reliable architecture suitable for mission-critical, telecom-grade messaging operations.

üîç Platform Overview

Before starting any SMPP-related service, it is mandatory to enter the SMPP Docker container. None of the services can run outside the container environment.

All services must be started in parallel. The platform begins by initializing the SMPP Server and SMPP Client.

The SMPP Server creates multiple listening ports defined in the .env file. Each port becomes active and ready to accept client connections.

The SMPP Client connects to the server using configured credentials (Username, Password, IP, Port). Once authenticated, the client is authorized to send and receive messages.

Invalid credentials result in immediate disconnection, ensuring security and controlled access.

1Ô∏è‚É£ SMPP_CLIENT Service

After successful connection, the SMPP Client prepares the message for delivery.

Required message parameters:

  • Username
  • Password
  • Source Address
  • Destination Address
  • Message Content

Messages are processed through the message action, where the channel is identified:

  • SMPP ‚Üí Channel = S
  • HTTP ‚Üí Channel = H
  • App / Website Campaign ‚Üí Channel = A

Based on data_coding, the message is passed to sendMessage(), where:

  • Message is split according to encoding rules
  • Extra TLVs are attached if required
  • Final payload is forwarded to the SMPP Server
2Ô∏è‚É£ SMPP_SERVER Service

On receiving a message, the SMPP Server performs:

  • Generation of a unique Message ID (OMSGID)
  • Reuse of client-supplied ID if provided
  • Revalidation of client credentials
  • Rate-limit and capacity checks

Messages are processed in per-second loops to maintain throughput and performance.

After validation, messages are forwarded to the DF Service.

3Ô∏è‚É£ DF Service (DragonFly)

The DF service receives the message payload from the SMPP Server and stores it securely in Redis (DragonFly).

  • Ensures persistence and queue safety
  • Stores metadata for downstream services
  • Acts as the backbone for message flow

After storage, the message is forwarded to the Validator Service.

4Ô∏è‚É£ Validator Service

The Validator Service retrieves message data from Redis and performs critical checks:

  • Billing completion verification
  • Destination blacklist checks
  • Source address blacklist checks
  • Spam and restricted content detection
  • TLV validation

Only messages that pass all validation steps are approved and forwarded to the Translation Service.

5Ô∏è‚É£ Translation Service

The Translation Service applies client-specific or message-specific transformation rules.

  • Fetch rules from Redis
  • Match rules against message/client
  • Modify or translate content if required

After translation, messages move to the Buffer Service.

6Ô∏è‚É£ Buffer Service

The Buffer Service controls message timing and delivery eligibility.

  • Check country & region restrictions
  • Validate promotional time windows
  • Hold messages outside allowed time periods
  • Queue messages with priority and retry rules

Eligible messages are forwarded to the FDLR Service.

7Ô∏è‚É£ FDLR Service

The FDLR Service applies controlled DLR splitting using cfdlr_ratio.

  • 100:0 ‚Üí All messages routed normally
  • Other ratios ‚Üí Partial real delivery + fake DLRs
  • Per-second ratio enforcement
  • Redis logging for reconciliation

Routed messages proceed to the Routing Service.

8Ô∏è‚É£ Routing Service

Routing rules are evaluated using client system_id and MCC/MNC data.

  • LCR ‚Äì Least cost supplier
  • PCR ‚Äì Premium supplier
  • CHOICE_LIST ‚Äì Percentage-based split

Supplier rates are fetched and messages are forwarded to the SEND_SMS Service.

9Ô∏è‚É£ SEND_SMS Service
  • Maintains supplier connections
  • Triggers billing before delivery
  • Sends provider-specific payloads
  • Receives and maps supplier DLRs

Standardized DLRs are forwarded to the DLR Service.
